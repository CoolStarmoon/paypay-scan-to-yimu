# 商户合集与用户分类映射需求文档

## 1. 背景

当前 App 会从 PayPay 使用记录截图中识别商户、金额和时间，并导出一木记账可导入的表格。现有分类逻辑写在代码中，适合单一用户，但不同用户的账本分类体系差异很大：

- 同样是便利店，有人归为“餐饮 / 零食”，有人归为“购物 / 日用品”。
- 同样是服装店，有人只有一级分类“购物”，不需要二级分类。
- 同样是订阅服务，有人归为“生活 / App会员”，有人归为“娱乐 / 数字服务”。

因此需要将“识别商户属于哪种消费场景”和“这个消费场景映射到用户账本中的哪个分类”解耦。

## 2. 目标

实现一套快速适配不同用户分类体系的分类机制：

1. App 内置商户合集规则，用于判断商户属于便利店、超市、服装店、交通、订阅服务等消费场景。
2. 用户首次使用时，为每个商户合集设置自己的一级分类和二级分类。
3. 二级分类允许为空。
4. 用户可以在识别结果中手动修正某个商户分类，App 记住该商户的单独规则。
5. 用户在首次设置完成后，仍可以从 App 内的设置入口再次进入分类规则设置并修改配置。
6. 分类规则支持导入和导出，便于备份、迁移和分享。
7. 导出时使用用户映射后的分类，而不是固定写死的个人分类。

## 3. 非目标

以下内容不作为第一版必须实现：

- 云端同步分类规则。
- 多设备共享配置。
- AI 自动推断用户账本分类体系。
- 在线商户库更新。
- 复杂条件规则编辑器，例如按金额区间、时间段、地点组合判断。
- 多账本配置切换。

## 4. 核心概念

### 4.1 商户合集 RuleGroup

商户合集表示一类具有相似消费语义的商户集合。它解决“这个商户是什么类型”的问题。

示例：

| group_id | group_name | 说明 |
| --- | --- | --- |
| convenience_store | 便利店 | Lawson、FamilyMart、7-Eleven 等 |
| supermarket | 超市 | Aeon、Valor、业务超市等 |
| restaurant_fast | 快餐 | McDonald's、KFC、Sukiya、Matsuya 等 |
| restaurant_dining | 正餐 | 寿司、烧肉、居酒屋、餐厅等 |
| clothing_store | 服装店 | Uniqlo、GU、Zara 等 |
| transport_train | 电车地铁 | JR、地铁、Suica、Pasmo 等 |
| subscription | 订阅服务 | OpenAI、Netflix、Spotify、Apple 等 |
| entertainment_movie | 电影娱乐 | TOHO、电影院、电影票等 |

每个商户合集包含商户名匹配关键词规则。

第一版内置商户合集需要覆盖足够多的常见商户名和 OCR 变体，避免只靠少量示例关键词导致匹配率过低。每个商户合集至少维护 100 个商户名匹配关键词，关键词可以包括：

- 商户英文名、日文名、中文常用名。
- 常见缩写、片假名、品牌别名。
- PayPay 记录中可能出现的收款方名称。
- OCR 容易识别出的高频变体。
- 同一品牌不同写法，例如空格、连字符、全角半角差异。

关键词库应当作为可维护的数据资产，而不是散落在代码逻辑中。新增、删除或修正关键词时，不应影响用户自己的分类映射。

示例 CSV：

```csv
group_id,group_name,keywords
convenience_store,便利店,"lawson|familymart|ファミマ|セブン|7-eleven|ローソン"
clothing_store,服装店,"uniqlo|gu|zara|無印|しまむら"
subscription,订阅服务,"openai|chatgpt|netflix|spotify|apple|google"
```

### 4.2 用户分类映射 CategoryMapping

用户分类映射表示某个商户合集在用户账本中应该写入哪个分类。它解决“这个类型在我的账本里叫什么”的问题。

示例：

| group_id | major | minor |
| --- | --- | --- |
| convenience_store | 餐饮 | 零食 |
| clothing_store | 购物 |  |
| subscription | 生活 | App会员 |
| transport_train | 交通 | 公共交通 |

其中 `minor` 可以为空。

### 4.3 商户单独修正规则 MerchantOverride

商户单独修正规则表示用户对某个具体商户的手动分类结果。它优先级最高。

示例：

| merchant_key | display_name | major | minor |
| --- | --- | --- | --- |
| lawson_shinjuku | LAWSON 新宿店 | 餐饮 | 早餐 |
| apple_com_bill | Apple.com/bill | 生活 | App会员 |

## 5. 分类匹配优先级

识别出一条账单后，按以下顺序决定分类：

1. 查询商户单独修正规则 `MerchantOverride`。
2. 根据商户名匹配内置或导入的商户合集 `RuleGroup`。
3. 根据合集 ID 查询用户分类映射 `CategoryMapping`。
4. 如果合集未映射，使用内置推荐映射。
5. 如果仍无法分类，标记为“待分类”。

流程：

```text
OCR 识别商户
 -> 商户名标准化
 -> 查 MerchantOverride
 -> 匹配 RuleGroup
 -> 查 CategoryMapping
 -> 使用 major/minor
 -> 导出到一木记账表格
```

## 6. 首次使用流程

### 6.1 触发时机

用户第一次打开 App 时，如果本地没有 `CategoryMapping` 配置，则进入首次分类设置流程。

### 6.2 设置内容

App 展示内置商户合集列表，每个合集需要用户设置：

- 一级分类，必填。
- 二级分类，可为空。

示例：

```text
便利店 -> 一级分类：餐饮，二级分类：零食
超市 -> 一级分类：餐饮，二级分类：食材
服装店 -> 一级分类：购物，二级分类：空
订阅服务 -> 一级分类：生活，二级分类：App会员
```

### 6.3 默认值

App 可以提供默认推荐分类，用户可以直接确认，也可以修改。

示例默认值：

| 商户合集 | 推荐一级分类 | 推荐二级分类 |
| --- | --- | --- |
| 便利店 | 食品餐饮 | 便利店 |
| 超市 | 食品餐饮 | 食材采购 |
| 快餐 | 食品餐饮 | 日常正餐 |
| 正餐 | 休闲娱乐 | 外出吃饭 |
| 服装店 | 购物消费 | 服饰鞋包 |
| 电车地铁 | 出行交通 | 公共交通 |
| 订阅服务 | 居家生活 | App会员 |
| 电影娱乐 | 休闲娱乐 | 电影娱乐 |

## 7. 后续分类规则设置入口

首次设置完成后，用户仍然需要能够再次进入分类规则设置，用于调整账本分类、修正导入结果或扩展个人规则。

### 7.1 入口位置

App 应提供稳定入口，例如：

- 设置页中的“分类规则设置”。
- 识别结果页中的分类相关菜单入口。
- 首次设置完成页或导入完成页中的“继续调整分类”入口。

第一版至少需要在设置页提供“分类规则设置”入口，其他入口可作为体验优化。

### 7.2 可编辑内容

分类规则设置页至少支持编辑：

- 商户合集到用户账本分类的映射，即 `CategoryMapping`。
- 商户单独修正规则，即 `MerchantOverride`。
- 每个合集的一级分类和二级分类，其中二级分类可为空。

第一版可以不支持在 UI 中编辑内置 `RuleGroup` 的关键词库，但需要让用户知道当前规则来源于内置合集或导入规则。若后续开放关键词编辑，应与导入导出的规则文件保持同一数据结构。

### 7.3 保存行为

用户修改分类规则后：

1. 新规则立即保存到本地配置。
2. 后续识别结果使用最新规则。
3. 当前页面已有识别结果时，应提示用户重新应用分类，或自动刷新分类结果。
4. 修改不会清空已有商户单独修正规则，除非用户明确删除。

## 8. 识别结果修正流程

识别完成后，每条账单展示：

- 商户名
- 日期时间
- 金额
- 当前一级分类
- 当前二级分类

用户可以点击某条账单修改分类。保存后：

1. 当前账单立即更新分类。
2. App 询问或默认保存为该商户的单独修正规则。
3. 以后遇到同名或标准化后相同的商户，优先使用该规则。

第一版可以采用默认保存策略，减少操作成本。

## 9. 导入与导出

分类规则需要支持导入和导出，用于用户备份、换设备迁移、重装 App 后恢复配置，以及在公开仓库或社区中分享通用规则。

导入导出入口应放在“分类规则设置”页中，避免只在首次启动流程中出现。

### 9.1 导入分类映射

支持从 CSV 导入用户分类映射。

格式：

```csv
group_id,major,minor
convenience_store,餐饮,零食
clothing_store,购物,
subscription,生活,App会员
```

导入后，如果 `group_id` 已存在，则更新该合集的用户分类映射；如果 `group_id` 不存在，应提示用户该行无法匹配内置或已导入的商户合集。

### 9.2 导入商户合集规则

支持从 CSV 导入商户合集规则，用于扩展或覆盖内置商户合集。

导入规则可以采用以下策略之一：

- 追加模式：保留内置关键词，在同一 `group_id` 下追加导入关键词。
- 覆盖模式：用导入文件中的 `group_id`、`group_name`、`keywords` 覆盖同名合集。

第一版建议默认使用追加模式，并在导入前展示影响范围，降低误覆盖风险。

格式：

```csv
group_id,group_name,keywords
convenience_store,便利店,"lawson|familymart|ファミマ|セブン"
```

每个导入后的商户合集仍应满足至少 100 个商户名匹配关键词。若导入文件导致某个合集关键词少于 100 个，App 应提示风险，但可以允许用户继续导入。

### 9.3 导入商户单独修正规则

支持从 CSV 导入商户单独修正规则。

格式：

```csv
merchant_key,display_name,major,minor
lawson_shinjuku,LAWSON 新宿店,餐饮,早餐
apple_com_bill,Apple.com/bill,生活,App会员
```

导入后，如果 `merchant_key` 已存在，则以导入文件为准覆盖原规则。

### 9.4 导出用户配置

用户可以导出自己的分类规则配置，用于备份或分享。

导出文件包括：

- `category_mapping.csv`
- `merchant_override.csv`
- `rule_groups.csv`

其中：

- `category_mapping.csv` 导出用户对商户合集的分类映射。
- `merchant_override.csv` 导出用户对具体商户的单独修正规则。
- `rule_groups.csv` 导出当前生效的商户合集规则，包括内置规则和用户导入扩展后的规则。

### 9.5 导入导出校验

导入时需要校验：

- CSV 表头是否符合预期。
- 必填字段是否为空。
- `major` 是否为空，一级分类不能为空。
- `minor` 可以为空。
- `keywords` 是否能被正确解析为关键词列表。
- 是否存在重复 `group_id` 或 `merchant_key`。

导出时需要保证：

- 文件使用 UTF-8 编码。
- 中文、日文、英文商户名不会乱码。
- 空二级分类导出为空字符串。
- 关键词顺序稳定，便于用户比较版本差异。

## 10. 本地存储

第一版建议使用 Android 本地文件或 SharedPreferences 保存配置，避免引入数据库复杂度。

推荐逻辑结构：

```text
rule_groups.json          当前生效的商户合集，包括内置合集和用户导入扩展
category_mapping.json     用户对合集的分类映射
merchant_override.json    用户对具体商户的手动修正
```

如果后续规则数量增多，再迁移到 SQLite 或 Room。

## 11. 商户名标准化

为了提高匹配率，需要对 OCR 识别出的商户名做标准化：

- 英文统一小写。
- 全角数字转半角数字。
- 去除多余空格。
- 去除常见无意义符号。
- 可选：去除店铺编号、分店号、支付后缀等。

示例：

```text
LAWSON 新宿三丁目店 -> lawson 新宿三丁目店
Apple.com/bill -> apple.com/bill
ファミリーマート 12345 -> ファミリーマート
```

## 12. 异常与兜底

### 12.1 未匹配合集

如果商户没有匹配到任何合集：

- 分类显示为“待分类”。
- 导出时一级分类可以写“待分类”，二级分类留空。
- 用户修正后保存为 `MerchantOverride`。

### 12.2 合集未设置映射

如果商户匹配到合集，但用户没有设置该合集的映射：

- 优先使用内置推荐映射。
- 如果没有推荐映射，则显示“待分类”。

### 12.3 二级分类为空

导出时二级分类单元格写空字符串。

## 13. 验收标准

第一版完成后应满足：

1. App 不再只依赖硬编码个人分类。
2. 内置至少 8 个商户合集：便利店、超市、快餐、正餐、服装店、电车地铁、订阅服务、电影娱乐。
3. 每个内置商户合集至少包含 100 个商户名匹配关键词。
4. 首次使用时可以为每个商户合集设置一级分类和二级分类。
5. 首次设置完成后，用户可以从设置页再次进入“分类规则设置”并修改规则。
6. 二级分类可以为空。
7. PayPay 截图识别结果能按用户映射输出分类。
8. 用户手动修改某个商户分类后，同商户再次出现时自动使用修改后的分类。
9. 用户可以导入和导出 `category_mapping.csv`、`merchant_override.csv`、`rule_groups.csv`。
10. 导入导出的 CSV 使用 UTF-8 编码，中文、日文、英文内容不乱码。
11. 导出的一木记账表格中，分类和二级分类来自用户映射或商户修正规则。
12. 未匹配商户不会导致导出失败，会进入“待分类”。

## 14. 推荐实施阶段

### 阶段一：分类核心解耦

- 新增 `RuleGroup`、`CategoryMapping`、`MerchantOverride` 数据结构。
- 将现有 `classify(merchant, amount)` 拆成规则匹配和用户映射。
- 内置第一批商户合集。
- 为每个内置商户合集整理至少 100 个商户名匹配关键词。

### 阶段二：首次设置向导

- 检测是否已有用户分类映射。
- 没有配置时展示合集映射表单。
- 保存用户设置。

### 阶段三：分类规则设置页

- 在设置页提供“分类规则设置”入口。
- 支持查看和修改 `CategoryMapping`。
- 支持查看、修改和删除 `MerchantOverride`。
- 修改后重新应用到后续识别结果。

### 阶段四：识别结果修正

- 在识别结果列表中支持修改分类。
- 保存商户单独修正规则。
- 重新渲染列表和导出结果。

### 阶段五：导入导出配置

- 支持导入 `category_mapping.csv`。
- 支持导入 `merchant_override.csv`。
- 支持导入扩展商户合集规则 `rule_groups.csv`。
- 支持导出 `category_mapping.csv`、`merchant_override.csv` 和 `rule_groups.csv`。

## 15. GitHub 开源视角

该设计适合公开仓库：

- 公开的是通用商户合集知识库。
- 用户自己的分类体系保存在本地配置中。
- 不需要把个人账本分类硬编码在源码里。
- 其他用户可以通过首次设置快速适配自己的账本。

仓库可以提供：

```text
docs/category-mapping-requirements.md
examples/category_mapping.example.csv
examples/rule_groups.example.csv
```
